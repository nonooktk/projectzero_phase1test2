# 役員向け資料 構成方針

> 想定読者：黒崎CDO（役員）。エンジニアではないため技術詳細は不要。
> 伝えるべき軸：「何ができるか」「なぜ必要か」「従来と何が違うか」

---

## 1. アーキテクチャ資料：「どう動くか」の図解

### 伝えるべき核心
「入力から答えまでの流れが自動化されている」

### フロー図（掲載レベル）

```
[① 入力]
  ユーザーがテーマを入力
  例：「ビルエネルギー管理で新事業を考えたい」
         ↓
[② データ検索]
  社内外の情報を自動で収集・整理
  ・市場・規制・競合（社外情報）
  ・自社技術・過去PJ・失敗事例（社内情報）
  ・関係しそうなキーマン（人物情報）
         ↓
[③ 3軸で自動評価]
  今やるべきか（外部環境）
  自社でやれるか（社内適合）
  誰とやるか（組織・人材）
  → 各軸に ◎○△× スコアと根拠を生成
         ↓
[④ GO/NO判定 + 事業案3つの生成]
  スコアに基づき推進可否を判定
  → 具体的な事業案3案 + 担当者アクション案
         ↓
[⑤ 承認者向けサマリーを自動出力]
  「なぜ GO/NO か」を1段落で要約
```

### 図に添える補足文（役員が気にするポイント）

| 補足すべき点 | 書き方の方向性 |
|---|---|
| 人手でやると何日かかるか | 「従来は担当者が数日かけて行っていた情報収集・整理を数十秒で実行」 |
| 判断の根拠が残る | 「スコアに根拠テキストが紐づくため、後から判断の経緯を確認できる」 |
| 失敗事例を参照している | 「過去の失敗条件が現在解消されているかを自動でチェックする仕組みがある」 |

## 2. バックエンド資料：意味検索 × 知識グラフの説明

> 技術名はサブタイトル扱い。ビジネス上の意義を前面に出す。

### 見出し1：「なぜ普通の検索ではダメなのか」

従来のキーワード検索では、「BEMS」と書いていないデータは見つけられない。
このシステムは「意味の近さ」で検索するため、表現が違っても関連情報を引っかけられる。

→ ここで初めて「意味検索（技術名：ChromaDB）」という言葉を出し、技術名は括弧書き程度に留める

---

### 見出し2：「なぜ関係者まで自動で見つかるのか」

市場や技術だけでなく、「この技術を担当できる人は誰か」「過去に関連PJを経験した社員は誰か」という
関係性を、あらかじめ社内の知識をネットワーク図として整理しておくことで自動的に辿れる。

→ グラフ構造を以下のような図で示す（実データから抜粋）

```
[BEMS市場]
   ├── 田村 浩二（営業ネットワークあり）
   ├── 薄膜太陽電池技術（応用できる）
   └── BEMS事業（過去PJ・失敗事例あり）
```

→ PyVis可視化の画面と連動させて説明すると効果的

---

### 見出し3：「2つを組み合わせることで何が変わるか」

| | 従来の社内検索 | このシステム |
|---|---|---|
| 検索方法 | キーワード一致 | 意味の近さで検索 |
| 関係者の特定 | 担当者が手動で探す | 知識グラフから自動で辿る |
| 失敗事例の参照 | 報告書を手動で調べる | 条件解消チェックを自動実行 |
| アウトプット | 情報の羅列 | GO/NOと根拠・事業案がセットで出力 |

## 3. 技術用語の言い換え一覧

| 技術用語 | 役員向けの言い換え |
|---|---|
| ChromaDB / ベクトルDB | 意味検索エンジン |
| Embedding / ベクトル化 | テキストを意味のデータに変換する処理 |
| GraphRAG / NetworkX | 社内知識のネットワーク図 |
| depth=1 の隣接ノード探索 | 1つ隣のつながりまでを自動で辿る |
| Stage1 / Stage2 | 個別評価フェーズ / 統合判断フェーズ |
| 並列処理 | 複数の評価を同時実行（省略可） |

---

## 4. 作成優先順位

1. **フロー図（①〜⑤）** → 「何ができるシステムか」を1枚で伝える。最優先
2. **従来との比較表** → 「なぜこのシステムが必要か」の根拠になる
3. **グラフ構造の図** → PyVis可視化の説明と連動させると効果的
