# システムアーキテクチャ 説明文書

---

## ワンメッセージ

**テーマを入力するだけで、数日かかっていた情報収集・整理・判断を数十秒で完了する。**

---

## 全体の処理フロー（5ステップ）

---

### ステップ1：ユーザー入力

ユーザーはフォームに3つの項目を入力する。

- ターゲット市場 / 想定顧客
- 活用したい自社アセット・コア技術
- 事業アイデアの詳細

この3つを1つのテーマ文字列として結合し、以降の処理に渡す。
PDF資料の添付も任意で行える。

---

### ステップ2：データ検索（ChromaDB による意味検索）

テーマ文字列を使って、ChromaDB（意味検索エンジン）で関連度の高いデータを上位5件取得する。

検索対象は以下の3種類のデータ。

| データ | 内容 |
|---|---|
| external.json | 市場・競合・規制情報 |
| internal.json | 社内技術・過去PJ・失敗事例 |
| persons.json | 社内キーマン情報 |

3種類のデータを「1つのデータベース（コレクション）」にまとめて管理しており、
「どのデータから来たか（source）」というタグで区別している。

**ポイント：キーワード検索との違い**
「BEMS」と書いていないデータでも、「ビルエネルギー管理」というテーマで
意味的に近いと判断されれば検索結果に含まれる。

---

### ステップ3：コンテキスト構築（2つの補完処理）

ステップ2の検索結果を元に、2つの独立した補完処理を行い、
最終的に3軸のコンテキストテキストを生成する。

**補完処理① ── グラフ検索（NetworkX）による関係者の補完**

検索ヒットしたデータのIDを起点に、ノード間のつながりを1ステップ辿る。
これにより、直接ヒットしなかった関係者・技術・市場ノードが自動で補完される。

例：「BEMS事業（past_project）」がヒット
→ グラフを辿ると「田村浩二（営業ネットワークあり）」が浮かび上がる
→ persons.json の検索では出てこなかった人物が補完される

**補完処理② ── internal.json 直接読込による失敗条件の補完**

ChromaDB に格納している `content` フィールドには入りきらない情報
（`conditions_now` / `reusable_assets` / `lessons_learned`）を
internal.json から直接読み込んでコンテキストに追記する。

これにより「過去の失敗条件が現在解消されたか」をAIが判断できるようになる。

**上記2つを統合して、以下の3軸コンテキストを出力する。**

- 外部コンテキスト（市場・競合・規制）
- 社内コンテキスト（技術・過去PJ・失敗条件）
- 組織コンテキスト（キーマン情報）

---

### ステップ4：AI分析（2段階・並列処理）

**Stage1（3並列）**

外部環境・社内適合・組織体制の3軸を、それぞれ独立したGPT呼び出しで
同時に分析する（3つを並列実行）。
各軸に ◎○△× スコアと評価根拠テキストを生成する。

**GO/NO判定（ルールベース）**

Stage1のスコアを以下のルールで機械的に判定する。人の主観は入らない。

| 条件 | 判定 |
|---|---|
| 社内適合スコアが × | NO（実行不可能な障壁あり） |
| 社内適合スコアが △ | 条件付きGO（障壁の解決を前提に検討可） |
| 外部環境スコアが △ または × | 条件付きGO（市場変化を確認しながら進める） |
| 全軸スコアが ◎ または ○ | GO（即時推進可） |

**Stage2（2並列）**

Tier1とTier2を同時実行する。

- Tier1：事業案3つ ＋ 承認者向けサマリーを生成
- Tier2：3C分析（Customer / Competitor / Company）を生成

---

### ステップ5：画面表示（Streamlit）

分析結果を以下の4エリアで表示する。

| 表示エリア | 内容 |
|---|---|
| 承認者サマリー | 「なぜGO/NOか」を1段落で自動要約（画面最上部） |
| 3軸評価 | ◎○△×スコア ＋ 評価根拠テキスト |
| 事業提案タブ | 事業案3つ ＋ 担当者アクション案 |
| 3C分析タブ | 市場・競合・自社の深掘り分析 |
| グラフタブ | PyVisによる知識グラフ可視化（検索ヒットノードをハイライト） |
